# 外部 HTTP 通知投递服务需求文档

## 1. 背景

企业内部存在多个业务系统，它们在关键业务事件发生后需要通知外部供应商系统，例如广告系统、CRM 系统、库存系统等。不同供应商的 HTTP(S) API 在请求地址、Header、Body 格式、认证方式和响应行为上都不一致。

如果每个业务系统都直接对接供应商 API，会带来以下问题：

- 业务系统需要重复处理重试、失败记录、格式适配等逻辑。
- 供应商接口不稳定时，会影响业务系统主流程。
- 缺少统一的通知投递状态查询、审计和补偿入口。
- 新增或替换供应商时，对业务系统改动较多。

因此需要一个内部通知投递服务，统一接收业务系统提交的外部 HTTP 通知请求，并尽可能可靠地投递到目标地址。

## 2. 目标

- 为内部业务系统提供统一的 HTTP 通知提交接口。
- 支持不同供应商的目标 URL、Header、Body 和 HTTP Method。
- 业务系统提交通知后无需等待外部供应商响应，可快速得到内部接收结果。
- 通知请求需要持久化，避免服务短暂异常导致任务丢失。
- 支持后台异步投递、失败重试、最终失败记录和状态查询。
- 提供简单的前端页面，用于演示通知提交和投递状态查看。

## 3. 非目标

- 不在本阶段实现供应商配置中心、模板编排或复杂字段映射。
- 不在本阶段实现分布式队列、分布式锁和多节点高可用部署。
- 不在本阶段实现 OAuth、签名算法、证书双向认证等供应商级认证方案。
- 不保证所有目标系统一定成功处理通知，只保证内部服务尽力可靠送达并记录结果。

## 4. 用户角色

- 业务系统：调用内部服务提交外部通知请求。
- 运维/业务支持人员：查看通知状态，必要时手动重新投递失败任务。
- 外部供应商系统：接收最终 HTTP 通知请求。

## 5. 核心场景

1. 用户通过第三方广告系统引流并成功注册后，业务系统提交通知，请求投递到对应广告系统。
2. 用户订阅付款成功后，业务系统提交通知，请求投递到 CRM 系统以更新 Contact 状态。
3. 用户购买商品后，业务系统提交通知，请求投递到库存系统进行库存变更。
4. 外部供应商暂时不可用时，服务自动重试，超过最大次数后记录为最终失败。
5. 支持人员可以查看任务状态，确认投递是否成功，并对失败任务发起手动重试。

## 6. 功能需求

### 6.1 提交通知

业务系统通过 `POST /api/notifications` 提交通知请求。

请求字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `targetUrl` | string | 是 | 外部供应商 HTTP(S) 地址 |
| `method` | string | 否 | HTTP Method，默认 `POST`，支持 `POST`、`PUT`、`PATCH` |
| `headers` | object | 否 | 投递到供应商时携带的 Header |
| `body` | object/string | 否 | 投递到供应商时发送的 Body |
| `eventType` | string | 否 | 事件类型，例如 `user.registered` |
| `sourceSystem` | string | 否 | 来源业务系统 |
| `requestId` | string | 否 | 业务侧幂等 ID，重复提交时返回同一任务 |
| `maxAttempts` | number | 否 | 最大投递次数，默认 5 |

响应示例：

```json
{
  "id": "6f7a6e93-2de3-47a6-b7cd-5be3da0b8482",
  "status": "queued",
  "duplicate": false
}
```

### 6.2 异步投递

- 服务接收请求后先写入数据库，状态为 `queued`。
- 后台 Worker 扫描可投递任务，将状态改为 `delivering`。
- Worker 根据任务中保存的 URL、Method、Header、Body 发起 HTTP 请求。
- 目标系统返回 2xx 视为投递成功。
- 非 2xx、连接失败、超时均视为本次投递失败。

### 6.3 失败重试

- 失败后，如果未达到最大次数，任务进入 `waiting_retry`。
- 下一次投递时间按指数退避计算。
- 达到最大次数后，任务进入 `failed`。
- 记录最近一次失败原因和目标系统状态码。

### 6.4 状态查询

支持查询通知列表和单个通知详情：

- `GET /api/notifications`
- `GET /api/notifications/{id}`

任务状态：

| 状态 | 说明 |
| --- | --- |
| `queued` | 已接收，等待投递 |
| `delivering` | 正在投递 |
| `waiting_retry` | 本次失败，等待下一次自动重试 |
| `succeeded` | 投递成功 |
| `failed` | 超过最大次数，最终失败 |

### 6.5 手动重试

支持通过 `POST /api/notifications/{id}/retry` 将任务重新入队。

## 7. 非功能需求

### 7.1 可靠性

- 通知任务必须先持久化再返回业务系统。
- 服务重启后，未完成任务可以继续被扫描和投递。
- 投递失败需要保留错误原因和尝试次数。

### 7.2 性能

- 提交通知接口应快速返回，不等待外部供应商接口响应。
- 本阶段演示服务使用单进程后台线程投递；生产环境可扩展为消息队列和多 Worker。

### 7.3 安全

- 生产环境应增加调用方鉴权，例如内部 API Key、mTLS 或网关注入身份。
- 生产环境应增加目标地址白名单，避免 SSRF 风险。
- Header 中的敏感值在日志和前端展示中应脱敏。

### 7.4 可观测性

- 记录任务创建时间、更新时间、投递时间、尝试次数、最近错误和目标状态码。
- 生产环境应接入日志、指标和告警，例如失败率、堆积量、投递延迟。

## 8. 数据模型

通知任务字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 内部任务 ID |
| `request_id` | 业务侧幂等 ID |
| `event_type` | 事件类型 |
| `source_system` | 来源系统 |
| `target_url` | 供应商地址 |
| `method` | HTTP Method |
| `headers_json` | 请求 Header JSON |
| `body` | 请求 Body |
| `status` | 当前状态 |
| `attempt_count` | 已尝试次数 |
| `max_attempts` | 最大尝试次数 |
| `next_attempt_at` | 下一次可投递时间 |
| `last_error` | 最近错误 |
| `last_status_code` | 最近目标状态码 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `delivered_at` | 成功投递时间 |

## 9. 演示实现范围

本仓库的示例实现包含：

- 一个 Python 标准库后端服务，无需安装第三方依赖。
- SQLite 持久化。
- 后台线程自动投递。
- 指数退避重试。
- 简单前端页面，支持提交通知、查看列表、查看状态和手动重试。
- 一个内置 mock 供应商接口：`/mock/vendor/{name}`，用于演示成功或失败投递。

## 10. 验收标准

- 可以通过页面或 API 提交通知。
- 提交成功后任务会持久化到 SQLite。
- 指向可用目标地址时，任务最终变为 `succeeded`。
- 指向失败目标地址时，任务会自动重试，达到最大次数后变为 `failed`。
- 可以查询任务状态和最近错误。
- 可以手动重新投递失败任务。

## 11. 后续演进

- 引入消息队列，例如 Kafka、RabbitMQ、Redis Stream 或云队列。
- 支持供应商模板配置和字段映射。
- 支持签名、认证、密钥托管和 Header 脱敏。
- 支持多租户和调用方权限控制。
- 支持死信队列、批量重试和运维告警。
- 支持更细粒度的投递日志，每次尝试单独记录。
