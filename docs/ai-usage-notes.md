# AI 使用说明

本文记录本项目中我与 AI 辅助工具的主要交互过程，以及我在实现过程中如何判断、筛选和修正 AI 的建议。

其中，“我”指项目实现者；“AI”指辅助需求拆解、代码实现、测试和文档整理的编码助手。

## 1. 交互步骤简要记录

1. 我先给出原始需求：企业内部多个业务系统需要把关键事件可靠地通知外部 HTTP(S) API，业务系统不关心外部 API 返回值，只要求通知尽可能可靠送达。
2. AI 帮助整理第一版需求文档，并生成一个简单的前后端服务：后端负责接收、持久化和异步投递通知，前端用于提交通知和查看状态。
3. 我询问项目文件所在目录、如何在页面白屏时调试，以及应该点击哪些按钮、看哪些状态来判断成功或失败。AI 帮助补充了本地运行链接、页面操作步骤、成功/失败测试用例和测试文档。
4. 我要求补充设计说明，尤其回答系统边界、可靠性语义、失败处理、取舍和未来演进。AI 帮助把这些内容整理到 README、设计文档和测试文档中。
5. 我要求将项目推送到 GitHub，并处理本地 Git 认证和仓库命名。AI 协助配置远端、提交代码、推送到 `rc_long_xueqian` 仓库。
6. 我要求使用多个子 agent 持续围绕原始需求迭代。AI 将工作拆成需求发现、代码编写、测试、文档和前端等角色，持续提出候选需求、实现小切片并做回归验证。
7. 在多轮迭代后，我要求暂停继续扩展，只完成当前轮并检查已有功能。AI 完成最后一轮 API Key 鉴权、文档补充和前端输入框，然后执行完整 smoke test、浏览器白屏验证、提交并推送。

## 2. AI 在关键地方提供的帮助

### 2.1 需求拆解和系统边界

AI 帮助把“可靠投递外部通知”拆成几个具体问题：

- 如何接收业务系统提交的通知请求。
- 如何持久化任务，避免提交后丢失。
- 如何异步投递，不阻塞业务系统主流程。
- 如何处理外部系统失败、超时和长期不可用。
- 如何让测试和排障看到投递过程，而不是只看到最终状态。
- 如何避免 Demo 变成任意内网探测代理或泄露敏感 Header。

这些拆解帮助我把第一版边界定在“内部通知投递层”，而不是做成完整集成平台。

### 2.2 代码实现和小步迭代

AI 参与实现了以下关键能力：

- 通知任务持久化、后台 Worker 投递、失败重试和状态更新。
- `notification_attempts` 投递尝试日志，用于查看每次 HTTP 调用结果。
- `requestId` 幂等提交，避免业务系统重复提交生成多条内部任务。
- URL 白名单、同源 mock vendor 例外、SSRF 基础拦截和敏感信息脱敏。
- 失败分类、超时配置、手动重试、批量重试、死信状态和人工处置字段。
- `/health`、`/api/stats`、分页、排序、时间范围过滤和 CSV 导出。
- 可选 `NOTIFICATION_API_KEYS` 调用方 API Key 鉴权。
- 前端提交表单、任务列表、详情抽屉、健康条、过滤控件、复制 curl 和 API Key 输入框。

AI 的帮助主要体现在快速生成初版代码、发现遗漏路径、把重复修改拆成较小的安全切片，以及每轮修改后补充对应测试。

### 2.3 测试和白屏调试

AI 帮助补充了两类测试：

- 自动化 smoke test：覆盖成功投递、失败重试、幂等、SSRF 拒绝、脱敏、重启恢复、死信、分页、CSV、timeout、API Key 鉴权等路径。
- 手工白屏调试文档：说明打开哪个本地链接、点击哪个按钮、观察哪些状态字段、如何判断投递成功或失败。

在最后验收时，AI 还通过浏览器验证页面没有白屏，健康条能显示 `serviceVersion`、Worker 存活状态和并发数，成功任务能显示“投递成功”，失败任务能显示“最终失败”。

### 2.4 文档和工程说明

AI 帮助把代码中的设计判断同步到文档中，包括：

- 需求文档。
- 设计说明。
- 测试计划。
- API 契约。
- API 示例。
- 多 agent 迭代计划。

这些文档让项目不仅“能跑”，也能解释为什么这样设计、哪些能力是第一版范围内的、哪些能力留给后续演进。

## 3. AI 给出过但我没有采纳的建议

| AI 或常见方案建议 | 是否采纳 | 原因 |
| --- | --- | --- |
| 直接引入 Kafka、RabbitMQ 或 Temporal | 未采纳 | 当前是作业和本地 Demo，重点是可靠性语义和失败处理。引入中间件会增加部署成本，反而不利于评审快速运行。 |
| 承诺“精确一次”投递 | 未采纳 | HTTP 通知在超时场景下无法判断对方是否已经处理。更真实的语义是“至少一次”，并要求下游做幂等。 |
| 一开始就做供应商配置中心和字段映射模板 | 未采纳 | 当前需求允许调用方提交完整 URL、Header 和 Body。第一版先验证投递链路，避免过早做复杂平台化抽象。 |
| 实现完整 RBAC、多租户、审计后台 | 未采纳 | 第一版只做可选共享 API Key 保护写接口。完整权限体系需要真实组织、租户和审计要求支撑。 |
| 直接实现 `/metrics` 和完整 Prometheus 接入 | 未采纳 | 当前只写指标草案和字段映射，不引入新依赖。第一版先通过 `/health` 和 `/api/stats` 满足排障。 |
| 对所有失败任务无限自动重试 | 未采纳 | 外部系统长期不可用时，无限重试会造成堆积、噪声和供应商压力。当前选择有限重试、失败状态、手动重试和死信处理。 |
| 把死信处理做成完整工单和审批流程 | 未采纳 | 投递服务只记录 dead letter 状态和人工备注，不替代供应商支持、审批、根因分析和审计系统。 |
| 把 Worker 并发当作主要扩容手段 | 未采纳 | SQLite 和单进程线程只适合本地 Demo 与小规模吞吐调整。未来增长应演进到 PostgreSQL、消息队列或调度系统。 |

## 4. 我自己做出的关键决策

### 4.1 选择“至少一次”投递语义

我决定采用至少一次，而不是最多一次或精确一次。

原因是通知系统的目标是“尽可能可靠送达”。如果选择最多一次，失败时可能丢消息；如果承诺精确一次，在 HTTP 超时和网络异常场景下又不真实。至少一次更符合 Webhook 和外部通知场景，但必须明确要求业务系统或供应商基于业务 ID、订单号、用户 ID 或 `requestId` 做幂等。

### 4.2 第一版使用 SQLite 和后台线程

我决定第一版不引入 Kafka、RabbitMQ、Celery、Temporal 等基础设施，而是使用 SQLite 和本地后台 Worker 线程。

原因是作业评估更关注工程判断和取舍逻辑，而不是组件堆叠。SQLite 加后台线程能清楚展示接收、持久化、投递、重试、查询和补偿的核心链路，也方便评审直接在本地启动。

### 4.3 先支持显式 HTTP 请求，不强制 vendorKey

我决定第一版让业务系统直接提交 `targetUrl`、`headers` 和 `body`，而不是强制使用供应商配置中心。

原因是原始需求强调不同供应商的地址、Header、Body 格式不同，但没有要求必须内置供应商管理。先让调用方显式提交请求可以最快验证可靠投递能力。`vendorKey` 作为未来演进方向记录在文档中，而不是作为当前必需功能。

### 4.4 保留本地 mock vendor，同时增加安全边界

我决定保留同源 `/mock/vendor/*` 用于演示和测试，但对其他 localhost、私网、metadata 地址做基础拒绝，并支持 `NOTIFICATION_ALLOWED_TARGETS` 白名单。

原因是本地 Demo 需要可重复验证成功和失败投递；但通知服务如果完全允许任意 URL，会变成 SSRF 风险点。这个折中既保证演示可运行，也体现了上线时必须考虑的安全边界。

### 4.5 失败后有限重试，再进入 failed 或 dead_letter

我决定不无限自动重试，而是记录失败原因、按最大次数重试，最终进入 `failed`，必要时由人工标记 `dead_letter`。

原因是外部供应商长期不可用时，无限重试不能解决根因，还会放大资源消耗和告警噪声。有限重试加人工补偿更容易控制，也更符合真实运维流程。

### 4.6 文档和测试与代码同步推进

我决定每一轮功能都同步补充测试和文档，而不是只写代码。

原因是这个题目不仅考察功能是否完成，也考察如何拆解问题、界定边界、识别复杂度和说明取舍。文档能展示判断依据；smoke test 和白屏调试说明能证明功能确实可验证。

### 4.7 最后一轮后停止继续扩展

在 AI 和子 agent 已经提出很多扩展点后，我决定最后一轮完成 API Key 鉴权后停止继续增加新需求，只做现有功能回归。

原因是继续扩展会增加代码和文档复杂度，也会提高回归成本。当前项目已经覆盖了原始需求的主路径、失败处理、排障、测试和设计说明，继续堆功能对第一版价值不大。

## 5. 对 AI 使用的总体判断

AI 在本项目中最有价值的地方，是帮助我快速生成候选方案、补齐测试矩阵、发现边界问题，并把代码、文档和验证流程串起来。

但关键工程判断仍需要我自己做：哪些能力进入第一版、哪些能力只写成演进方向、哪里需要控制复杂度、哪里不能为了“看起来完整”而过度设计。

因此，本项目不是简单照搬 AI 输出，而是把 AI 当作需求拆解、编码和测试助手；最终方案仍围绕可运行、可解释、可验证和复杂度可控这几个目标收敛。
