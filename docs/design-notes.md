# API 通知系统设计说明

## 1. 系统边界

### 1.1 本系统选择解决的问题

本系统定位为“内部业务系统到外部 HTTP(S) API 的通知投递层”，第一版重点解决以下问题：

- 统一入口：业务系统只需要调用内部 `POST /api/notifications`，不用在各自系统里重复实现投递、重试和状态记录。
- 异步解耦：业务系统提交成功后即可继续自己的主流程，不等待外部供应商接口返回。
- 请求适配：支持不同供应商的 `targetUrl`、HTTP Method、Header 和 Body。
- 持久化：通知任务先写入 SQLite，再返回业务系统，降低服务重启或进程异常导致任务丢失的风险。
- 自动重试：外部系统临时失败时，后台 Worker 自动按指数退避重新投递。
- 状态查询：提供任务列表、单任务详情、错误原因和每次投递尝试记录，方便排查。
- 手动补偿：支持对失败任务重新入队，也支持把最终失败任务标记为 `dead_letter`，记录最近一次人工动作和处置说明。
- 最小安全边界：支持可选调用方 API Key、目标 Origin 白名单、同源 mock vendor 例外、基础 SSRF 拦截和展示脱敏，避免 Demo 变成无保护写入口或任意内网探测代理。
- 演示闭环：内置 mock 供应商接口，方便验证成功投递和失败重试。

这些能力覆盖了题目中的核心诉求：业务系统不关心外部 API 返回值，只希望通知请求能被尽可能可靠地送达。

### 1.2 本系统明确不解决的问题

第一版明确不解决以下问题：

- 不做供应商配置中心和字段映射模板。  
  原因：当前需求允许业务系统直接提交目标 URL、Header 和 Body。第一版先验证投递链路，避免过早引入配置平台、模板 DSL、版本管理等复杂度。后续可以用 `vendorKey` 演进出供应商配置能力，但不作为当前创建通知的必填字段。

- 不做“精确一次”投递。  
  原因：HTTP 通知天然很难保证精确一次。网络超时场景下，本系统无法判断目标系统是否已经处理请求。强行宣称精确一次反而不真实。第一版采用至少一次投递语义，并要求下游通过业务幂等键处理重复通知。

- 不判断外部系统的业务处理结果。  
  原因：本系统只把 2xx 作为 HTTP 层投递成功。供应商返回 200 但业务内部处理失败的场景，需要供应商 API 契约或回调机制支撑，不适合作为第一版内部投递层职责。

- 不内置复杂认证和签名。  
  原因：不同供应商可能使用 API Key、HMAC、OAuth、mTLS 等不同方案。第一版允许业务系统透传 Header，生产化时再增加密钥管理和签名插件。

- 不做外部死信队列、工单系统和完整人工处置流程。
  原因：第一版只在通知投递层提供 `dead_letter` 状态、人工动作字段和显式恢复入口，用来冻结自动投递并保留交接信息。供应商侧工单、根因分析、权限审批、审计合规和 SLA 追踪需要组织流程和外部系统配合，不适合塞进这个投递服务的第一版。

- 不做分布式高可用、跨进程并发锁和消息队列。
  原因：当前是作业和演示场景，使用单进程后台线程、SQLite 和 `DB_LOCK` 更容易运行和理解。第一版可以在单进程内通过线程数提高慢 HTTP 等待时的吞吐，但不承诺跨实例消费和分布式调度能力。

- 不做完整安全治理。  
  原因：当前只提供可选的 `NOTIFICATION_API_KEYS` 共享 Key 来保护写接口，并已经实现目标地址白名单、基础 SSRF 防护和敏感信息展示脱敏；它仍不包含用户身份、权限分级、密钥轮换审计和重定向目标复检。第一版先覆盖最容易出事故的未授权写入、目标地址和展示泄密风险；重定向目标复检已经作为下一层安全契约和测试项，生产化必须补上。

死信/人工处理的边界尤其需要明确：本系统只回答“这条通知是否还应继续由投递层自动发送、最近一次人工处置是谁做的、备注是什么”。它不判断供应商为什么长期失败，也不替代供应商支持工单、业务根因分析、审批流、权限复核或审计合规系统。未来如果人工处置量明显上升，可以把 `dead_letter` 任务推送到工单系统，并把工单编号回写到 `resolutionNote` 或扩展字段中。

## 2. 可靠性与失败处理

### 2.1 投递语义

本系统选择“至少一次”投递语义。

含义：

- 只要任务被成功写入本系统数据库，本系统会尽力投递到目标地址。
- 如果投递失败，会按照策略重试。
- 在超时、服务重启、人工重试等场景下，同一条通知可能被投递多次。
- 业务系统或外部供应商需要基于业务 ID、订单号、用户 ID、事件 ID 或 `requestId` 做幂等处理。

为什么选择至少一次：

- “最多一次”会降低重复风险，但失败时可能丢通知，不符合“尽可能可靠送达”的目标。
- “精确一次”在 HTTP 外部系统通知场景中成本很高，也依赖目标系统配合，第一版不适合承诺。
- “至少一次”是通知、事件投递、Webhook 这类系统中更常见且务实的语义。

### 2.2 接收阶段的可靠性

业务系统提交通知后，本系统先将任务写入 SQLite，然后再返回响应。

这样做的判断依据：

- 如果先返回再写库，服务进程异常会导致业务系统以为提交成功但任务丢失。
- 写库后返回可以把任务状态固定下来，后续即使外部系统失败，也能查询和补偿。

当前实现支持 `requestId`，用于业务侧提交幂等：

- 如果业务系统重复提交同一个 `requestId`，服务返回已存在的任务。
- 这能降低业务系统重试提交时产生重复内部任务的概率。
- 这不能替代外部供应商侧幂等，因为投递阶段仍可能多次发送 HTTP 请求。

### 2.3 投递阶段的失败处理

外部系统失败时，当前策略如下：

- 目标系统返回非 2xx：视为本次投递失败。
- 连接失败或超时：视为本次投递失败。
- 未达到最大尝试次数：状态进入 `waiting_retry`。
- 达到最大尝试次数：状态进入 `failed`。
- 每次投递尝试都会写入 attempts 记录，包括尝试编号、本次结果、HTTP 状态码、耗时、错误和时间。
- 任务本身同时保留最近错误原因和目标 HTTP 状态码，方便列表快速判断。
- 支持人工点击 `重新投递`，将任务重新入队。

重试策略：

- 使用指数退避。
- 当前示例从 5 秒开始，随后按倍数增长，上限为 300 秒。
- 每条任务可配置 `maxAttempts`，示例默认 5，最大限制 10。

为什么不无限重试：

- 如果供应商长期不可用，无限重试会造成任务堆积和资源浪费。
- 长期失败通常需要业务或运维判断，例如供应商配置错误、认证失效、合同接口变更。
- 第一版选择“有限自动重试 + 明确失败状态 + 人工补偿”，更容易解释和控制。

### 2.4 外部系统长期不可用时的策略

当前第一版：

- 自动重试到最大次数。
- 进入 `failed` 后保留错误原因。
- 支持页面或 API 手动重试。
- 支持把 `failed` 任务标记为 `dead_letter`：该状态停止自动投递，也不会被普通批量重试选中；只有单条人工 retry 才能重新入队。
- 服务启动时回收上一次进程退出前遗留的 `delivering` 任务，避免重启后任务永久卡住。
- Worker 运行中按 `NOTIFICATION_DELIVERING_LEASE_SECONDS` 回收超出租约的 `delivering` 任务，并记录 `lease_timeout` attempt。

生产演进时：

- 将当前 `dead_letter` 标记演进为更完整的死信队列或死信表，支持负责人、截止时间、工单编号和处置结果。
- 支持批量重试、按供应商重试、按事件类型重试，但对死信任务保持显式人工确认，避免误操作。
- 增加供应商级熔断和限流，避免对已故障供应商持续施压。
- 增加告警，例如失败率升高、堆积量升高、投递延迟升高。
- 增强 attempts 的分页、失败分类、人工重试批次编号和审计字段，方便大规模排障。
- 将 delivering 租约回收从单 worker loop 演进为独立 watchdog 或多 Worker 协作。如果唯一 Worker 线程永久卡在系统调用里，独立 watchdog 能更稳定地回收超时任务；租约仍应大于正常 HTTP 超时，避免误回收正在正常执行的慢请求。

### 2.5 业务提交失败与供应商投递失败

本系统刻意区分两类失败，避免业务系统把“已被通知服务接收”和“供应商已经处理成功”混为一谈。

| 失败类型 | 发生阶段 | 是否已经创建任务 | 典型状态/响应 | 处理策略 |
| --- | --- | --- | --- | --- |
| 业务系统提交失败 | 调用 `POST /api/notifications` 时 | 否 | `400` 参数错误、`500` 服务异常、网络错误 | 业务系统按响应修正或重试；没有任务状态可查 |
| 业务系统重复提交 | 调用 `POST /api/notifications` 时 | 是 | `200` 且 `duplicate=true` | 视为提交成功，返回已有任务 ID |
| 供应商投递失败 | Worker 调用外部 API 时 | 是 | `waiting_retry`、`failed`、`failureType`、attempts | 业务主流程不阻塞；由本系统自动重试、记录和人工补偿 |

这个划分的判断依据是：业务系统关心的是“我的通知事件有没有被内部服务可靠接收”；供应商投递结果属于异步交付过程，应通过状态查询、管理页、告警或补偿流程处理。

### 2.6 死信与人工处理策略

`dead_letter` 是第一版系统对“自动重试已经不适合继续”的显式状态，而不是另一个自动重试队列。

使用场景：

- 供应商接口长期不可用，继续重试只会制造噪声。
- 供应商地址、认证或字段格式明显配置错误，需要先修复配置。
- 业务方已经通过线下方式补偿，不希望系统再自动发送重复通知。
- 需要把失败任务交给值班、业务或供应商支持人员处理。

状态策略：

- `failed` 可以标记为 `dead_letter`。
- `dead_letter` 不再被 Worker 自动领取。
- 普通批量重试默认只处理 `failed`，不处理 `dead_letter`。
- 单条 `POST /api/notifications/{id}/retry` 可以把 `dead_letter` 重新入队，这是明确的人工恢复动作。
- 正在 `delivering` 的任务不能标记死信，避免投递线程仍在执行时状态被强行覆盖。

人工字段：

- `lastManualAction` 记录最近动作，例如 `dead_letter` 或 `retry`。
- `lastManualActionAt` 记录动作时间。
- `lastManualActionBy` 记录操作者或系统标识。
- `resolutionNote` 记录处置说明，但不应写入密码、token、客户隐私等敏感信息。

这样设计的判断依据是：投递服务需要提供“停止自动化、保留人工交接信息、允许显式恢复”的最小闭环，但不应该替代供应商侧工单和组织内审批审计流程。

### 2.7 Worker 并发配置

`NOTIFICATION_WORKER_CONCURRENCY` 用来配置后台投递 Worker 的线程并发数。文档约定默认值为 `1`，也就是单 Worker 线程；建议范围为 `1` 到 `4`。如果后端实现设置了更高硬上限，可以在启动时 clamp 或拒绝非法值，但第一版不建议把它当作无限扩容开关。

适合调大的信号：

- `/health.queue.readyCount` 长时间大于 0，说明已到投递时间的任务没有被及时领取。
- 任务主要卡在外部 HTTP 等待，CPU、数据库写入和本机网络仍有余量。
- 外部供应商允许更高并发，或者已经按供应商做了限流和超时控制。
- 下游已经按业务幂等键、订单号或 `requestId` 处理重复通知。

调大的风险：

- SQLite 写入、任务领取和状态更新仍受 `DB_LOCK` 串行保护，并发线程越多，等待锁和上下文切换越明显。
- 对同一个慢或故障供应商并发投递可能放大 429、5xx、超时和告警噪声。
- 租约回收、手动重试和至少一次语义本来就允许重复投递；并发越高，边界条件下的重复窗口越需要下游幂等兜底。
- 单进程线程数不能解决进程退出、机器故障、SQLite 文件锁或多实例消费问题。

因此第一版的定位是“单机吞吐微调”，不是生产级队列系统。并发调大前应先看 `/health.worker.concurrency`、`worker.threadCount`、`worker.aliveCount`、`queue.readyCount` 和供应商错误率；如果 `aliveCount` 小于配置值，或 `readyCount` 仍持续增长，就应排查线程异常、数据库瓶颈或供应商限流，而不是继续盲目加线程。

## 3. 取舍与演进

### 3.1 第一版没有采纳的“过度设计”

在常见 AI 方案建议或完整企业级方案中，可能会直接引入很多能力。对这个作业场景，我认为以下设计第一版偏过度，因此没有采纳：

- 直接引入 Kafka 或 RabbitMQ。  
  这些中间件适合生产流量和多实例消费，但会显著增加本地运行成本。当前 Demo 的重点是说明投递语义和失败处理，SQLite 加后台线程已经能表达核心设计。

- 直接做供应商配置中心和模板编排。  
  字段映射、模板版本、供应商密钥、灰度发布都是真实需求，但第一版会让问题从“可靠投递”变成“集成平台”。本系统先让调用方提交完整 HTTP 请求。

- 直接做工作流引擎。  
  Temporal、Airflow、Celery 这类工具适合复杂编排和长流程，但本场景当前只有“接收、投递、重试、完成”这条短链路。

- 直接做多租户权限体系。  
  内部服务最终需要鉴权和租户隔离，但 Demo 阶段只做可选共享 API Key 保护写接口，不把大量篇幅放在用户身份、RBAC 和租户模型上。

- 直接实现所有供应商认证方式。  
  OAuth、HMAC、mTLS 都可能需要，但应该在真实供应商接入时按需扩展，而不是第一版预置一套难以验证的抽象。

- 直接做完整可观测性平台。  
  生产需要日志、指标、链路追踪和告警。当前代码记录核心状态字段即可，避免把作业演示变成基础设施搭建。

判断依据是：第一版应该覆盖核心风险，同时保持可运行、可解释、可验证。过早引入分布式组件和平台化抽象，会掩盖题目真正关注的工程边界和可靠性语义。

更具体地说，我对 AI 方案建议做了如下取舍：

| AI 常见建议 | 第一版决策 | 判断依据 |
| --- | --- | --- |
| 上来就用 Kafka、RabbitMQ 或 Temporal | 不采纳 | 当前任务量和评审环境未知，本地零依赖更能证明可靠性语义；中间件会把复杂度转移到部署和运维 |
| 把供应商做成配置中心、模板 DSL 和字段映射平台 | 暂不采纳 | 真实供应商契约尚未固定，先允许业务系统传完整 HTTP 请求，避免抽象先行 |
| 实现精确一次投递 | 不承诺 | HTTP 超时无法判断供应商是否已处理，强行保证精确一次不真实；采用至少一次加业务幂等 |
| 所有失败都无限重试 | 不采纳 | 长期不可用时会造成堆积和重复压力；有限重试后进入 `failed`，再人工或批量补偿 |
| 失败后直接接入完整工单系统和审批流 | 暂不采纳 | 第一版只做 `dead_letter` 标记和人工字段，先把投递层状态边界说清楚；工单、审批、根因和合规审计需要外部流程支撑 |
| 做完整 RBAC、多租户、审计后台 | 暂不采纳 | 内部 Demo 只用可选共享 API Key 保护写接口；生产接入多业务方后再补身份、权限和审计 |
| 所有供应商认证方式都内置插件 | 暂不采纳 | OAuth、HMAC、mTLS 的实现取决于具体供应商；当前允许透传 Header，后续按真实接入扩展 |
| 把前端做成完整运营平台 | 暂不采纳 | 第一版页面目标是白屏调试、状态查看和手动补偿，不追求复杂运营功能 |
| 完全忽略安全边界，只做可投递 | 不采纳 | 即使是 Demo，也不能变成内网探测代理；保留 URL 白名单、mock 例外和 SSRF 基础拦截 |

### 3.2 当前实现的主要取舍

| 设计点 | 当前选择 | 原因 | 代价 |
| --- | --- | --- | --- |
| 队列 | SQLite 表 + 后台线程扫描 | 零依赖，便于演示和本地运行 | 单机能力有限，不适合高并发生产 |
| 投递语义 | 至少一次 | 更符合可靠通知目标 | 可能重复投递，需要下游幂等 |
| 重试 | 有限次数 + 指数退避 | 控制失败成本，便于观察最终状态 | 长期失败需要人工或运维补偿 |
| 成功判断 | HTTP 2xx | 简单、通用 | 无法保证供应商业务处理成功 |
| 供应商适配 | 调用方传 URL/Header/Body | 灵活，第一版少抽象 | 业务系统仍需知道供应商格式 |
| 安全边界 | 可选调用方 API Key + Origin 白名单 + SSRF 拦截 + 展示脱敏 | 覆盖最直接风险，仍保持 Demo 简单 | 未覆盖用户身份、RBAC、密钥轮换审计和 redirect 复检 |
| 前端 | 简单管理页 | 方便演示和调试 | 不适合作为生产运营后台 |

### 3.3 调用方 API Key 鉴权取舍

`NOTIFICATION_API_KEYS` 是第一版对“内部写入口不能裸奔”的最小补强。默认不设置时关闭，保持本地 Demo、smoke test 和评审环境容易启动；一旦设置，`POST /api/notifications`、单任务 retry、dead-letter 和批量 retry 都必须带 `X-Notification-Api-Key` 或 `Authorization: Bearer ...`。

这个设计解决的是最窄的一件事：只有知道共享 Key 的内部调用方或运维脚本才能创建或修改通知任务。它不试图回答“这个人是谁”“他属于哪个租户”“他是否只能操作某个供应商”“这次操作是否经过审批”。因此第一版不引入用户表、角色表、租户模型、后台登录态、权限表达式或审计流水。

保护范围刻意只覆盖写接口：

- 创建通知、手动重试、标记死信和批量重试会改变队列状态，需要 Key。
- 列表、详情、attempts、CSV、`/health`、`/api/stats`、静态页面和 mock vendor 仍保留为调试入口，方便白屏排查和本地验收。
- 读接口仍可能包含脱敏后的业务排障信息；生产化时应根据数据敏感度把只读接口也纳入网关、登录态或只读权限控制。

这个边界也不解决供应商侧鉴权。payload 中的 `headers.Authorization` 或 `X-Vendor-Token` 是 Worker 投递给供应商的凭证；调用方提交给通知服务的 `X-Notification-Api-Key` / Bearer Key 只用于进入通知服务写入口，不能被写入任务、attempts、日志、截图或工单说明。

未来演进路径：

- 把共享 Key 托管到密钥系统，支持多 Key 命名、启停、过期时间和轮换窗口。
- 在网关层引入 JWT、mTLS 或内部服务身份，替代裸共享 Key。
- 将调用方身份绑定到 `sourceSystem`、`vendorKey`、允许事件类型和操作范围。
- 给 retry、dead-letter、批量 retry 等人工动作增加操作者身份、审批和审计日志。
- 对读接口和 CSV 导出增加权限控制与导出审计，避免排障入口变成批量数据泄露入口。

### 3.4 vendorKey 的取舍和未来设计

当前第一版让业务系统直接提交 `targetUrl`、Header 和 Body。这个选择有两个好处：

- 需求验证快：广告、CRM、库存三个场景的差异可以直接通过请求体表达，不需要先建设配置后台。
- 责任边界清楚：通知服务先证明“接收、持久化、投递、重试、查询”这一条链路可靠，而不是一开始就变成供应商集成平台。

它的代价也很明确：

- 多个业务系统可能重复填写同一个供应商 URL 和 Header。
- 供应商地址或认证信息变更时，需要通知多个调用方同步改配置。
- 调用方手填 `targetUrl` 容易误投到错误环境，例如把测试 CRM 地址带到生产请求。
- 安全白名单只能约束 Origin，不能表达“某个业务方只能调用某几个供应商”的业务边界。

未来可以引入 `vendorKey`，但它应该是流量和供应商复杂度上来后的演进，而不是第一版必备能力。目标形态示例：

```json
{
  "vendorKey": "crm-prod",
  "eventType": "subscription.paid",
  "sourceSystem": "billing-service",
  "requestId": "billing:subscription.paid:S-20260609-001",
  "body": {
    "contactId": "C-10086",
    "status": "paid"
  }
}
```

服务端配置示例：

```json
{
  "crm-prod": {
    "targetUrl": "https://crm.vendor.example/api/contacts/status",
    "method": "PATCH",
    "defaultHeaders": {
      "AuthorizationSecretRef": "secret://crm/prod/token"
    },
    "timeoutSeconds": 3,
    "maxAttempts": 5,
    "allowedEventTypes": ["subscription.paid"],
    "allowedSourceSystems": ["billing-service"]
  }
}
```

`vendorKey` 的价值不是少传几个字段，而是把供应商配置、密钥、白名单、限流、重试、超时和权限绑定到一个受控配置上。这样能降低重复配置和误投风险，也能把敏感 Header 从业务系统请求中移走。

第一版暂不实现 `vendorKey` 的判断依据是：

- 目前还没有真实供应商契约和密钥管理系统，强行设计配置模型容易变成空泛抽象。
- 直接透传请求更容易用 mock vendor 验证可靠投递和失败处理。
- 现有 `requestId`、`eventType`、`sourceSystem`、`targetUrl` 字段已经为将来迁移到 `vendorKey` 保留了审计和过滤基础。

### 3.5 单任务 timeoutSeconds 的取舍

全局 `NOTIFICATION_DELIVERY_TIMEOUT_SECONDS` 适合作为默认投递超时，因为它让运维可以用一个环境变量控制大多数供应商请求的最大等待时间。当前默认值是 `8.0` 秒，环境变量非法时回退默认值，合法数值限制在 `0.1` 到 `60.0` 秒。

Round K 增加的任务级 `timeoutSeconds` 是对这个默认值的窄覆盖，而不是新的重试策略。业务系统创建通知时可以传入该字段；省略或传 `null` 时，任务不保存覆盖值，Worker 投递时继续使用当前进程的全局配置。传入有限数字时，服务把值限制在 `0.1` 到 `60.0` 秒并持久化到任务；布尔值、非数字字符串、对象、数组或非有限数字会让创建接口返回 `400`，避免把明显错误的配置写进队列。

适合使用任务级超时的场景：

- 某个供应商明确要求快速失败，例如 CRM 更新必须在 2 秒内返回，否则由后续补偿处理。
- 某类通知不应长时间占用 Worker，例如低价值或高频事件。
- 某个供应商接口调用成本高，慢响应继续等待只会放大队列积压。

不建议滥用任务级超时的场景：

- 为每个请求随意传不同值，导致排障时无法判断系统真实默认行为。
- 用超短超时代替供应商限流、熔断或重试策略。
- 把超时时间设得接近 `NOTIFICATION_DELIVERING_LEASE_SECONDS`，使租约回收和正常慢请求边界变模糊。租约时间应明显大于正常投递超时。

这个设计保留了两层控制：全局默认解决“绝大多数供应商应该等多久”，任务级字段解决“这条通知或这个供应商确实不同”。未来迁移到 `vendorKey` 后，`timeoutSeconds` 更适合收敛到供应商配置中，由通知服务统一管理，而不是让每个业务系统长期手填。

### 3.6 列表分页、排序和导出的取舍

第一版列表查询选择 `limit + offset` 分页，而不是 cursor 分页。

Round J 增加的 `createdFrom`、`createdTo`、`updatedFrom`、`updatedTo` 继续沿用同一套列表查询模型。它们解决的是人工排障中最常见的“只看某个创建/更新时间窗口内任务”的问题，并让 CSV 导出可以精确复现页面筛选条件。时间范围端点采用闭区间，非法时间返回 `400`，这样调用方能尽早发现脚本或页面参数错误，而不是误以为空结果代表没有任务。

这样选择的原因：

- 调试直观：白屏排查和人工验收时，`offset=0&limit=20`、`offset=20&limit=20` 很容易理解，也方便直接用 curl 或浏览器地址栏复现。
- 实现简单：当前存储是 SQLite，列表主要服务于 Demo 管理页和小规模排障，offset 分页能用少量代码覆盖过滤、排序和导出。
- 契约透明：页面分页、时间范围过滤、CSV 导出和 API 调试可以共用同一组 query 参数，便于评审时观察行为。
- 数据量假设明确：第一版不是生产运营后台，通知量和历史保留规模都较小，offset 的性能问题暂时不是主要风险。

这个选择的代价也很清楚：

- 当数据量很大时，大 offset 查询会越来越慢。
- 在不断有新任务写入的情况下，用户翻页可能遇到重复或漏看的记录，尤其是按 `createdAt desc` 查看最新任务时。
- 如果导出大量历史数据，同步 CSV 会占用请求线程、数据库连接和浏览器下载时间，不适合生产大批量导出。

因此第一版应把 offset 分页视为“低流量、人工调试、同步导出”的方案。为了降低风险，列表排序需要使用稳定兜底字段，例如排序字段相同后再按 `id` 或创建时间排序；导出接口也要受 `limit` 和服务端最大值约束，不能无限制导出全部历史。

未来数据量增长后，演进方向是：

- 为常用过滤和排序建立索引，例如 `(status, created_at)`、`(source_system, created_at)`、`(event_type, created_at)`、`(target_url, created_at)`，并保留主键兜底排序。
- 将高频列表从 offset 演进为 cursor 分页，例如使用 `(createdAt, id)` 或 `(updatedAt, id)` 组成不透明 cursor，保证“下一页从上一页最后一条之后继续”。
- 管理页仍可保留 offset 作为小范围调试能力，但核心运营列表和 API 消费方优先使用 cursor。
- 大批量 CSV 从同步请求演进为异步导出任务：提交导出条件后生成 export job，后台分批扫描并写文件，完成后返回下载链接和过期时间。
- 对导出任务增加权限、审计、大小限制和脱敏校验，避免把排障工具变成批量泄密入口。

### 3.7 是否使用开源中间件

当前 Demo 没有引入额外开源中间件或外部基础设施。

当前选择：

- 使用 Python 标准库实现 HTTP 服务和后台 Worker。
- 使用 SQLite 做本地持久化。
- 使用 `DB_LOCK` 串行化 SQLite 读写、任务领取和状态更新，避免同一进程内多个线程同时改写文件数据库。
- 使用 `NOTIFICATION_WORKER_CONCURRENCY` 做单进程线程并发配置，默认 `1`，只用于缓解慢 HTTP 等待导致的本地吞吐不足。

为什么这样选择：

- 本地启动成本最低，评审或使用者不需要安装消息队列、Redis、Docker 或额外依赖。
- 作业重点是系统边界、可靠性语义、失败处理和取舍逻辑，不是中间件运维能力。
- SQLite 能表达“先持久化再返回”“重启后继续处理”的核心思想。
- 线程 Worker 足以表达异步投递、重试、租约回收和健康观测；`DB_LOCK` 让第一版的并发控制集中、可读，避免为了 Demo 引入分布式锁、broker ack、consumer group、visibility timeout 等一整套运维概念。
- Kafka、RabbitMQ 或 Celery 会把问题从“通知投递语义是否清楚”转移到“broker 如何部署、监控、持久化、升级和排障”，对第一版评审和本地运行成本偏高。

如果生产化，会考虑引入：

- RabbitMQ：适合任务队列、确认机制和重试/死信队列，运维复杂度中等。
- Redis Stream：适合轻量队列和内部系统，部署简单，但可靠性和消费语义要谨慎设计。
- Kafka：适合高吞吐事件流和多消费者，但对这个第一版通知系统来说初始成本偏高。
- Celery 或 RQ：适合 Python 后台任务，但需要 Redis 或 RabbitMQ 作为 broker。
- 调度系统或工作流引擎：适合复杂延迟调度、批量补偿、人工审批和跨服务编排，但只有在通知链路变成多步骤流程后再考虑。

不使用中间件时的替代方案：

- 当前 SQLite 扫描表方案可以支撑 Demo 或很小规模内部系统。
- 生产如果仍不使用消息队列，至少需要使用支持行级锁的数据库，例如 PostgreSQL，通过 `SELECT ... FOR UPDATE SKIP LOCKED` 实现多 Worker 抢占任务。
- 演进路径可以先从 SQLite + `DB_LOCK` 到 PostgreSQL 行锁 + 多进程 Worker，再按堆积量、跨实例需求和供应商限流复杂度迁移到 RabbitMQ/Kafka/Celery/调度系统；迁移时保留 `requestId` 幂等、attempts 审计、租约/可见性超时和死信处置语义。

## 4. 未来演进路线

如果这是第一版系统，未来流量或复杂度显著增长，可以按风险逐步演进。

### 4.1 流量增长

触发信号：

- 通知任务堆积明显。
- 平均投递延迟升高。
- 单进程 Worker 处理不过来。
- SQLite 写入和扫描成为瓶颈。
- 供应商接口开始要求不同限速，单一 Worker 扫描无法公平处理各供应商队列。

演进方向：

- 将 SQLite 替换为 PostgreSQL 或消息队列。
- 将后台 Worker 拆成独立进程，支持水平扩容。
- 按供应商、事件类型或租户分队列。
- 增加并发投递数、限流和超时配置。
- 引入任务可见性超时或租约机制，确保 Worker 异常时任务能被其他 Worker 重新领取。
- 为列表过滤和排序增加组合索引，并将大 offset 分页逐步替换为基于 `(createdAt, id)` 或 `(updatedAt, id)` 的 cursor 分页。

### 4.2 供应商复杂度增长

触发信号：

- 多个业务系统重复填写相同供应商 URL 和 Header。
- 供应商接口频繁变更。
- 不同供应商需要不同签名、认证和字段转换。

演进方向：

- 增加供应商配置表。
- 增加请求模板和字段映射。
- 增加密钥管理和 Header 脱敏。
- 增加供应商级超时、重试次数、限流和熔断配置。
- 增加供应商级幂等键映射规则，保证同一个内部 `requestId` 能转换为供应商要求的幂等 Header 或 Body 字段。
- 将 `targetUrl`、默认 Header、供应商密钥、允许事件类型和调用方权限收敛到 `vendorKey` 配置，减少业务方重复填写和误投。
- 支持从“透传 targetUrl”渐进迁移到“优先 vendorKey”：早期两者并存，生产核心供应商只允许 vendorKey，调试或低风险场景保留显式 targetUrl。

### 4.3 运维要求增长

触发信号：

- 失败任务需要批量处理。
- 业务方需要审计每一次投递尝试。
- 需要 SLA、告警和值班处理。

演进方向：

- 增强投递尝试日志的分页、检索、导出和人工重试批次标识。
- 将 `dead_letter` 标记接入工单系统，支持工单编号、负责人、处置 SLA、关闭原因和回写状态。
- 增强批量重试能力，但对死信任务保留显式确认和操作审计。
- 接入 Prometheus、Grafana、日志平台和告警。
- 增加任务搜索、过滤、导出和权限控制。
- 增加错误分类聚合看板，按供应商、状态码、失败类型和来源系统定位问题。
- 将大批量 CSV 导出改为异步导出任务，后台生成文件，前端轮询导出状态，完成后再下载，并记录导出审计。

Prometheus 文本指标在 Round K 只作为演进草案记录，不在当前代码中实现 `/metrics`，也不引入 Prometheus 客户端依赖。原因是当前 Demo 仍以 Python 标准库和零依赖为边界，`/health` 与 `/api/stats` 已经能覆盖第一版页面展示、人工排障和轻量统计；而 Prometheus 接入还需要确定 scrape 拓扑、实例标签、指标命名、标签基数和告警规则。

草案指标可以从现有 JSON 映射：`/health.database.ok` 对应 `notification_database_up`，`/health.worker.alive` 对应 `notification_worker_alive`，`queue.counts` 对应 `notification_queue_status_total{status}`，`queue.readyCount` 对应 `notification_queue_ready_total`，`/api/stats.attempts.recentErrorsByType` 对应 `notification_attempts_recent_errors_by_type_total{error_type}`。未来真正落地时，要禁止 `requestId`、`targetUrl`、用户 ID、原始错误文本这类高基数或敏感字段进入指标标签。

Round J 的 `serviceVersion` 和 `schemaVersion` 是轻量排障字段，主要用于判断“新前端是否打到了旧服务”或“页面是否按错误响应结构解析”。它们不等同完整迁移系统：不能替代数据库 migration、发布编排、灰度兼容矩阵或回滚策略。第一版只要求 `/health` 和 `/api/stats` 暴露足够信息，让白屏和错配能被快速定位。

详情抽屉复制 `requestId` 和 `targetUrl` 也是调试取舍的一部分。`requestId` 让支持人员能把前端详情、业务事件、供应商幂等日志和后端 attempts 串起来；`targetUrl` 让白屏或过滤异常时可以直接核对页面展示值、API 详情值和 Network 请求目标。复制的是排障入口，不改变投递语义，也不绕过目标地址白名单和脱敏边界。

### 4.4 安全要求增长

触发信号：

- 允许更多业务系统接入。
- 目标地址不再完全可信。
- Header 或 Body 中出现敏感信息。

演进方向：

- 将 `NOTIFICATION_API_KEYS` 从共享 Key 演进到可管理的调用方身份，例如网关 JWT、mTLS 或服务身份。
- 扩展写接口鉴权到只读接口、CSV 导出和管理页，按数据敏感度提供只读/操作权限。
- 将当前目标 Origin 白名单和 SSRF 防护生产化，包括重定向目标复检、DNS rebinding 防护和供应商配置绑定。
- 扩展当前敏感 Header 和 Body 字段脱敏展示，增加权限控制和查看审计。
- 增加审计日志，记录谁提交、谁重试、谁查看。

### 4.5 演进顺序建议

如果把当前实现作为第一版，我会按风险而不是按“功能看起来完整”的顺序演进：

1. 先补生产安全底线：把共享 API Key 演进为可轮换、可审计的调用方身份，供应商 Origin 白名单配置化，重定向目标复检，供应商密钥托管。
2. 再补运维可见性：失败率、队列堆积、投递延迟、Worker 心跳和告警。
3. 当任务堆积或多实例需求出现时，再把 SQLite Worker 替换为 PostgreSQL 行锁或消息队列。
4. 当供应商数量明显增加时，再做供应商配置、模板映射、签名插件和供应商级限流。
5. 当人工补偿频率上升时，再补死信队列、批量操作审计和更细的失败聚合报表。

这样演进的依据是：每一步都由真实风险或容量信号触发，而不是因为系统设计图上“应该有”。这能让第一版保持清晰，也能给后续扩展留下稳定接口。

## 5. 总结判断

这个第一版系统选择把主要复杂度放在“可靠接收、异步投递、失败重试、状态可见”上，而不是一开始就平台化。

核心判断是：

- 对业务系统而言，最重要的是主流程不被外部供应商拖慢，并且通知任务不会无声丢失。
- 对外部 HTTP 通知而言，至少一次比精确一次更现实。
- 对第一版工程实现而言，可运行、可验证、边界清晰，比功能铺满更重要。
- 对未来演进而言，当前的数据模型和状态机保留了替换队列、扩展 Worker、增加供应商配置和接入告警的空间。
